## 0. 待确认问题（会影响 S2 起的实现，建议先拍板）

### Q1. "买卖点"不是规则计算结果，而是 LLM 产出 —— task.md S7.3 的假设需要修正

`analysis_service._build_analysis_response` 里的 `strategy.ideal_buy/secondary_buy/stop_loss/take_profit`
读的是 `result.get_sniper_points()`，其实现（`src/analyzer.py:1840`）是：

```python
def get_sniper_points(self) -> Dict[str, str]:
    if self.dashboard and 'battle_plan' in self.dashboard:
        return self.dashboard['battle_plan'].get('sniper_points', {})
    return {}
```

`dashboard` 是 LLM 生成的报告结构（见 `src/analyzer.py`），**不是** `TrendAnalyzer` 的输出。
我在 `src/core/pipeline.py:2222-2237` 找到的唯一"规则兜底"，只覆盖 `stop_loss` 一项，且逻辑极简单：

```python
def _stop_loss_fallback_from_trend(trend_result, report_language):
    levels = getattr(trend_result, "support_levels", None) if trend_result else None
    if levels:
        return levels[0]          # 取 support_levels 第一个值
    return "待补充"
```

`ideal_buy` / `secondary_buy` / `take_profit` **没有任何规则兜底**，纯靠 LLM。
而 task.md S7 明确要求 `RuleScoreEngine` 是纯函数（无网络、无 LLM）。

**结论：** S7.3 不存在"对齐 pipeline 语义"这回事——Python 侧本来就没有纯规则版的买卖点算法。
端上要做，就是**新设计**一套基于 `support_levels/resistance_levels/MA/ATR` 的规则买卖点，
而不是"移植"。建议：
- 方案 A（推荐）：仅规则化 `stop_loss`（= 最近支撑位，与 Python 兜底逻辑一致）与
  `take_profit`（= 最近压力位），`ideal_buy/secondary_buy` 用"回踩 MA5/MA10 且乖离率在阈值内"
  这类已有的 `bias_ma5` 逻辑重新定义（复用 `_calculate_bias` 的判断区间：<-3%/-5%/+2%/阈值），
  在 Prompt 里明确写"以下买卖点为端上规则估算，非 AI 建议"。
- 方案 B：MVP 阶段 `ideal_buy/secondary_buy/take_profit` 直接置空/`not_supported`，只给 `stop_loss`。

### Q2. 两套 RSI 计算口径互相矛盾，需要选一个

- `technical_indicators.py::calculate_rsi`：`avg_gain/avg_loss` 用 **简单滚动均值**
  `rolling(window=period, min_periods=1).mean()`（SMA 口径）。
- `src/stock_analyzer.py::_calculate_rsi`：用 **Wilder's EMA/SMMA**
  `ewm(alpha=1/period, adjust=False).mean()`。

两者对同一段日线会算出**不同的 RSI 数值**。task.md 把两者都列为迁移对象
（S6.1 对应 `technical_indicators.py`，S7.1/7.2 对应 `stock_analyzer.py`），
但没说端上只保留一套。另外 `_calculate_mas`（stock_analyzer.py）用
`rolling(window=period)`（默认 `min_periods=period`，不足周期为 NaN），
而 `calculate_ma`（technical_indicators.py）用 `min_periods=1`（不足周期也出值）——
两者 MA 在数据不足 20/60 根时行为也不同。

**结论：** 端上 `TechnicalIndicators`（S6）与 `RuleScoreEngine`（S7）如果分别对拍两个 Python 源，
数值会对不上。建议**统一采用 `stock_analyzer.py` 的口径**（Wilder RSI + 满窗 MA），
因为 `RuleScoreEngine` 的评分权重直接依赖这一版本；`technical_indicators.py` 的
`calculate_all_indicators/get_latest_signals` 仅作为"信号淡出/展示"层的参考实现，
对拍时以 `stock_analyzer.py` 为准。也请确认这个取舍。

### Q3. `code` 归一化规则与既有 `Shared/Models/StockSymbol.swift` 不一致

现有共享层 `StockSymbol.init?(code:)`（已被 Watch/自选股使用）只处理
`6→sh, 0/3→sz, 4/8→bj`，其余（含 `5`、`9` 开头）返回 `nil`。
`WatchlistViewModel.add` 的报错文案也写死"须以 0/3/4/6/8 开头"。

但 Python 侧真实规则（`data_provider/base.py::is_bse_code`）更细：

```python
def is_bse_code(code: str) -> bool:
    # 900xxx 是上交所 B 股，必须排除
    if c.startswith("900"):
        return False
    return c.startswith(("92", "43", "81", "82", "83", "87", "88"))
```

且 `plan.md` 本任务文档 §2.1 与 task.md §S2.3 都要求"沪 6/5/9→sh，京→bj，其余→sz"——
即 `5`（如 `510300` 等 ETF/基金代码）、`9`（如 `900xxx` B 股）也要能落到 `sh`，
这在现有 `StockSymbol` 里完全没有支持。

**结论：** task.md S2.3 要求的 `canonical_stock_code` 归一化，**不能复用**现有
`Shared/Models/StockSymbol.swift` 的判断逻辑（也不应该改它——S1.1 明确"不改动现有自选股/Watch代码"）。
需要在 `Workspace/` 里新写一个独立的 code 归一化函数（例如 `StockCodeNormalizer`），
覆盖 `92/43/81/82/83/87/88`（且排除 `900`）→ `bj`，`6/5/9`（非 BSE）→ `sh`，其余 → `sz`，
与 `StockSymbol` 各自独立、互不影响。已在 S1 的占位文件里预留了这个文件位置，
具体判断表在 S2.3 落地时再写。

### Q4. 实时行情覆盖当日日线需要 `volume`，但现有 `Quote` 模型没有这个字段

Python `_augment_historical_with_realtime`（`src/core/pipeline.py:2361`）覆盖当日 K 线时，
连 `volume`/`amount`/`pct_chg` 一起替换：

```python
df.loc[idx, 'close'] = price
if vol: df.loc[idx, 'volume'] = vol
if amt is not None: df.loc[idx, 'amount'] = amt
```

而现有 `Shared/Models/Quote.swift` 只有 `price/previousClose/open/high/low/changeAmount/changePercent`，
**没有 volume/amount**；`TencentQuoteProvider.parse` 也没有解析这两个字段（腾讯 `~` 分隔数组里
第 6/37 位左右通常是成交量/成交额，现有解析器直接跳过了）。

**结论：** S5.2"实时覆盖"要做到和 Python 语义一致（覆盖 close 且更新 volume），
要么 (a) 扩展一个端上专用的"日线实时覆盖"数据结构（不改 `Quote` 本身，只在
`Workspace/` 内新增一个更完整的实时字段模型），要么 (b) MVP 先只覆盖
`close/open/high/low`，`volume` 维持日线原值（对量能类因子/评分有轻微失真，
建议在 ContextPack 的 `warnings` 里注明）。两种都可行，推荐 (b) 先做，减少
对既有 `QuoteProvider` 协议的改动面。

## 2. 数据源字段核对（对应 S0.2）

### 2.1 日线接口 `qfqday` 字段序（来自 `data_provider/tencent_fetcher.py::_extract_kline_rows`，已逐行核实）

```
row[0] = date（字符串 YYYY-MM-DD）
row[1] = open
row[2] = close     ← 注意：第 3 位是 close，不是 high！顺序是 open, close, high, low
row[3] = high
row[4] = low
row[5] = volume（单位：手）
row[6] = amount（可选，可能缺失）
```

**这个顺序容易搞反**——常见预期是 `open, high, low, close`，但腾讯接口实际是
`open, close, high, low`。Swift 新增的日线 Provider（S5.1）解析时务必按此顺序取值，
并对 `row.count < 6` 做防御（沿用现有 `TencentMinuteProvider` 的防御式解析风格）。

- 端点：`https://web.ifzq.gtimg.cn/appstock/app/fqkline/get`，
  `param={sh|sz|bj}{code},day,{start},{end},{lookback},qfq`
- 成交量单位换算：`_lots_to_shares`：`股数 = 手数 × 100`（简单浮点乘法，无特殊取整）
- 与现有 Swift 对齐：`TencentMinuteProvider.swift` 已实现同源分时数据解析（`minute/query` 端点），
  日线是姊妹端点，同一域名 `web.ifzq.gtimg.cn`，可复用其 `fetchJSON`/防御式解析风格，
  但 JSON 结构不同（`data.{symbol}.qfqday` 而非分时的 `data.{symbol}.data.data`），需要新写解析函数。

### 2.2 实时行情覆盖逻辑核对（来自 `src/core/pipeline.py::_augment_historical_with_realtime`，已读全函数）

- 非交易日直接跳过覆盖（`is_market_open(market, market_today)` 为 False 时原样返回）
- 若日线最后一行日期 `>= today`：**原地覆盖**该行的 `close/open/high/low/volume/amount/pct_chg`
  （`open` 优先取 `realtime.open_price`，缺失则退到 `pre_close`，再退到昨收）
- 若日线最后一行日期 `< today`：**追加一条虚拟当日行**（字段同上，`volume` 缺省 0）
- 详见 Q4：Swift `Quote` 模型当前不带 volume/amount，需要在这里做一个取舍
